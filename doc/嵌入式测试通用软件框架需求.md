通用软件测试平台需求
目标
兼容Seeed主流硬件平台（Nordic，ESP，CM4，CM5，RK，Jetson，TI...），底层由 SE 封装好，并向上提供 API 给 TE/EE 调用 标准的 测试命令，实现无差别的板子硬件功能验证

想要解决的问题
1.避免一款款板子写测试固件
2.EE / TE 新同事，大多对软件代码不熟悉，免去他们学习软件的的过程

需求只需要针对抽象层，具体实现不需要考虑
# 设计目标：

- **模块化设计**：各组件独立，通过明确定义的接口交互

- **状态驱动**：所有测试状态通过中间结果文件传递

- **可扩展架构**：新增测试用例无需修改核心框架

- **实时监控**：仪表盘提供可视化的测试进度和状态

- **资源隔离**：测试用例之间相互独立，避免影响

# 高内聚低耦合：
    单一职责原则 (SRP): 确保每个类/模块只做一件事。
    模块化设计: 将系统拆分为功能专一的模块。
    使用接口和抽象: 模块间不直接依赖实现类，而是依赖接口。
    依赖注入 (DI): 通过配置方式管理对象依赖，减少硬编码。
    合理划定边界: 梳理模块间关系，归类相似功能

所有的测试功能组件必须有的设置：
-h 使用说明，-I 循环次数，-i 间隔时间，-r 开启输出报告，-w 超时时间 ；缺少必须的参数需要提示报错


框架思路：由最小的功能组件【functions】（单个测试函数，带返回值）-> 组成测试用例【cases】（初始化，测试，根据json设置进行判断{pass/failed}输出report）-> 多个测试用例组成测试治具【fixtures】（由模板建立，通过config.json配置由哪些cases组成）

1.需要通过config.json文件设置开启哪些功能及设置相关参数：
a.功能测试只需要单次测试，EMS，EMI，高低温需要循环测试，
eg:高低温即由多个对应的测试用例组成高低温测试治具，一键运行
b.失败要有重试机制，重试次数和间隔在config文件或者调用时传参
	
2.各功能测试结果需要有log保存，研发测试时最好能保存系统log日志（如RK）：
注：方便异常时EE，SE可以查看日志排查是硬件还是系统调度问题。

3.各functions,cases需要有对应的Readme说明文档，（TE补充，需要AI提示词模板，代码永google注释风格，有脚本可以直接提取成wiki）
4.循环测试场景下需要有结果看板
  【CPU温度（类似于jtop，如图参考）；功能测试结果（需要自适应添加内容-数据驱动/模式发现）】

5.主要以shell为主，但需要能兼容py，需要留出拓展调用接口，接收返回值。
6.接口定义：功能状态码：
状态码	说明
0	测试成功
1	超时
2	缺少参数（缺少必要参数/带选项【option】没带参数）
-1	测试失败，未满足条件
-101	找不到设备
-102	设备异常
-103	未找到指定路径文件
	
-2	软件缺少环境/包

嵌入式测试通用软件框架需求文档
一、项目背景
1.1 角色定义
●SE（Software Engineer）：软件工程师，负责底层驱动封装和API开发
●TE（Test Engineer）：测试工程师，负责执行测试和验证
●EE（Electrical Engineer）：硬件工程师，参与硬件相关测试
1.2 核心痛点
1.重复开发效率低下：每款板子都需要重新编写测试固件
2.技术门槛高：EE/TE新同事大多对软件代码不熟悉，学习成本高
3.缺乏标准化：不同平台测试方式不统一，难以规范流程
1.3 项目目标
●兼容Seeed主流硬件平台：Nordic、ESP、CM4、CM5、RK、Jetson、TI等
●API解耦：底层由SE封装，向上提供标准API给TE/EE调用
●无差别测试：实现统一的测试命令，屏蔽底层硬件差异

二、框架架构设计
2.1 三层架构（模块化设计）

┌─────────────────────────────────────────┐
│  Functions（最小功能组件）               │
│  └── 原子性测试功能，可复用的最小单元    │
├─────────────────────────────────────────┤
│  Cases（按模块分的测试用例）             │
│  └── 每个硬件模块的完整测试逻辑          │
├─────────────────────────────────────────┤
│  Fixtures（按场景分的测试治具）          │
│  └── 组合多个Cases，对应具体测试场景     │
└─────────────────────────────────────────┘

2.1.1 Functions（最小功能组件）
●定义：单个原子测试功能，带返回值
●特点： 
○原子性：一个function只做一件事
○独立性：可以单独运行和测试
○可复用：可被多个Cases调用
●实现：使用类Linux的shell命令方式实现
●返回：标准状态码（详见4.1）
●示例：
# Functions示例
test_eth_basic --ip 192.168.1.100
test_eth_speed --ip 192.168.1.100 --mode iperf3
test_wifi_connect --ssid Seeed_Test
test_usb_detection

2.1.2 Cases（按模块分的测试用例）
●定义：每个硬件模块的完整测试逻辑
●组织方式：按硬件模块划分（不是按场景划分）
●结构： 
○模块名称（如eth_case, wifi_case, usb_case）
○包含该模块的完整测试流程
○初始化（Initialization）
○测试执行（Test Execution，调用多个Functions）
○结果判断（Result Validation）
○输出报告（Report Generation）
●配置：每个Case有独立的config.json
Case示例：eth_case.json

{
  "case_name": "eth_module_test",
  "description": "以太网模块完整测试",
  "module": "network",
  "functions": [
    {
      "name": "test_eth_basic",
      "params": {"ip": "192.168.1.100"},
      "enabled": true
    },
    {
      "name": "test_eth_speed",
      "params": {"ip": "192.168.1.100", "mode": "iperf3"},
      "enabled": true
    },
    {
      "name": "test_eth_led",
      "params": {},
      "enabled": true
    }
  ],
  "execution": "sequential",
  "timeout": 60
}

2.1.3 Fixtures（按场景分的测试治具）

●用途：一键运行复杂测试场景，无需手动组合Cases
●配置：每个Fixture有独立的config.json，指定包含哪些Cases

2.1.4 架构优势
✅ 模块化设计（高内聚、低耦合）
●Functions：原子功能，职责单一
●Cases：模块完整测试，逻辑内聚
●Fixtures：场景组合，灵活复用
✅ 避免重复测试
●每个模块的测试逻辑集中在一个Case里
●Fixture只负责选择需要哪些模块Cases
●不会出现重复测试同一模块
✅ 易于维护和扩展
●修改模块测试：只改对应的Case文件
●新增模块：创建新的Function + Case
●新增场景：创建新的Fixture组合已有Cases
●互不影响
✅ 灵活的执行控制
●Cases可配置：串行/并行执行
●Fixtures可配置：串行/并行执行、循环测试、重试机制
●满足不同场景需求

六、日志与报告系统
6.1 日志系统架构（四级输出）
6.1.1 设计原则
●分离显示：不同场景不同输出，互不干扰
●分级调试：3级debug等级，满足不同需求
●实时监控：中间结果文件供看板实时读取
●完整追溯：详细日志保存，便于问题排查
6.1.2 四级输出目标
标准的日志存放规范：
1.系统日志 ： /var/log/ - Linux系统的标准日志目录
2.应用日志 ： /var/log/应用名/ - 应用程序的专用日志目录
3.临时日志 ： /tmp/ - 临时日志文件
不建议放在项目文件夹的原因：
1.影响Git仓库 ：日志文件会使仓库变得臃肿，影响克隆和推送速度
2.不需要版本控制 ：日志文件是运行时生成的，不需要纳入版本控制
3.可能包含敏感信息 ：日志中可能包含系统信息或敏感数据
4.不符合Linux标准 ：违反了Linux系统的文件系统布局规范
