## 1. 设计目标与范围

### 1.1 设计目标
1. 统一 Seeed 主流硬件平台（Nordic、ESP、CM4/CM5、RK、Jetson、TI）的测试调用方式。
2. 提供标准化命令与状态码，降低 EE/TE 使用门槛。
3. 通过 Functions → Cases → Fixtures 三层结构实现高复用、低耦合。
4. 支持研发与生产两类场景：单次测试、循环测试、生产全检、抽检。
5. 提供四级输出体系：终端、详细日志、中间结果、报告，并支持可视化看板。
6. 支持一键部署与环境自更新。

### 1.2 范围（In Scope）
- 抽象层架构与接口规范。
- 配置模型、执行引擎、日志报告、Dashboard 设计。
- SOC（Linux）与 MCU（Zephyr）的统一调用抽象。
- MCP 服务层与 AI Agent 集成。

### 1.3 非范围（Out of Scope）
- 具体硬件驱动实现细节。
- GUI 复杂交互产品化设计（仅 CLI Dashboard 必须、Web Dashboard 可选）。
- 云端平台完整实现（仅定义上传接口与契约）。

## 2. 总体架构设计

### 2.1 分层架构

```text
┌──────────────────────────────────────────┐
│              AI Agent Layer              │
│     Skill / MCP Client / JSON Gen       │
├──────────────────────────────────────────┤
│             MCP Service Layer            │
│   MCP Server / Tools / Resources        │
├──────────────────────────────────────────┤
│           Interface Layer                │
│  CLI/API（run_fixture、run_case、函数命令） │
├──────────────────────────────────────────┤
│            Orchestration Layer           │
│  Fixture Runner / Case Runner / Scheduler│
├──────────────────────────────────────────┤
│             Test Logic Layer             │
│  Functions（原子能力）+ Case 编排逻辑     │
├──────────────────────────────────────────┤
│          Platform Adapter Layer          │
│  LinuxAdapter / ZephyrAdapter / CommonIO │
├──────────────────────────────────────────┤
│           Output & Observability         │
│  terminal + logs + tmp + reports + dash  │
└──────────────────────────────────────────┘
```

### 2.1.1 层级说明
| 层级                   | 职责                             | 关键组件                    |
| ---------------------- | -------------------------------- | --------------------------- |
| AI Agent Layer         | 自然语言交互、意图理解、配置生成 | Skill、MCP Client           |
| MCP Service Layer      | 标准化 API 暴露给 AI、自动发现   | MCP Server、Tools           |
| Interface Layer        | CLI 命令入口、参数解析           | run_fixture、run_case       |
| Orchestration Layer    | 测试编排、串并行调度             | FixtureRunner、CaseRunner   |
| Test Logic Layer       | 原子测试函数、模块编排           | Functions、Cases            |
| Platform Adapter Layer | 硬件平台适配                     | LinuxAdapter、ZephyrAdapter |
| Output & Observability | 日志、报告、监控                 | Logger、Dashboard           |

### 2.2 与需求三层的映射
- Functions：最小可复用原子测试单元。
- Cases：按模块定义完整测试逻辑（初始化/执行/判断/输出）。
- Fixtures：按场景组合多个 Cases，支持串并行、循环、重试、失败策略。

### 2.3 架构关键原则
- 单一职责：Function 不承担编排，Case 不承担跨场景编排。
- 配置驱动：所有行为由 JSON 配置控制。
- 状态驱动：执行状态统一写入 `tmp/*_result.json`。
- 可扩展：新增模块仅需新增 Function + Case，不改核心调度器。

## 4. 核心模块设计

### 4.1 CLI 接口层

#### 4.1.1 命令入口
- `run_fixture --name <fixture> --config <path> [--sn <SN>]`
- `run_case --config <case_json>`
- `test_xxx`（Function 命令）

#### 4.1.2 标准参数规范
所有 Function 必须支持以下标准参数：

| 参数 | 全称              | 说明           | 默认值 |
| ---- | ----------------- | -------------- | ------ |
| `-h` | `--help`          | 显示使用说明   | -      |
| `-I` | `--loop-count`    | 循环次数       | 1      |
| `-i` | `--loop-interval` | 循环间隔（秒） | 0      |
| `-r` | `--report`        | 开启报告生成   | false  |
| `-w` | `--timeout`       | 超时时间（秒） | 60     |

**参数校验规则**：
- 缺少必要参数返回 `2`（MISSING_PARAM）并提示错误
- 参数格式错误返回 `2` 并提示正确格式
- 帮助参数 `-h` 优先于其他所有参数

#### 4.1.3 装饰器实现
使用 `@test_function` 装饰器统一注入标准参数能力，设计约束如下：

- 入参：支持 `required_params` 声明必填参数集合。
- 注入能力：统一支持 `help`、循环、报告、超时等标准参数。
- 校验行为：在 Function 执行前完成必填校验、类型转换与默认值填充。
- 返回约束：参数错误统一返回状态码 `2`，并提供可读错误信息。
- 可追溯性：参数解析与校验过程写入调试日志（按日志级别控制）。

#### 4.1.4 参数传递机制
```
CLI 参数 (--ip 192.168.1.1 -I 5 -w 30)
    ↓
argparse 解析
    ↓
装饰器校验（必需参数、类型转换）
    ↓
Function 执行
    ↓
结果返回（状态码 + 详情）
```

### 4.2 调度引擎（Scheduler）
职责：
1. 加载 Fixture/Case 配置。
2. 根据 `execution` 决定串行/并行。
3. 执行循环、重试、失败处理（`stop_on_failure`）。
4. 维护运行上下文（loop_idx、retry_count、start_time、SN/SKU）。

### 4.3 资源隔离策略

#### 4.3.1 资源分类
| 资源类型 | 示例                     | 访问模式  | 隔离策略 |
| -------- | ------------------------ | --------- | -------- |
| 独占资源 | /dev/ttyUSB*, /dev/ttyS* | 串行      | 互斥锁   |
| 共享资源 | 网络接口 eth0            | 并行      | 无锁     |
| 条件共享 | I2C 总线                 | 并行 + 锁 | 总线锁   |

#### 4.3.2 资源锁设计
新增 `config/resources.lock` 管理资源互斥关系，配置模型约束如下：

- `exclusive_resources`：定义独占资源集合。
    - 关键字段：`name`、`pattern`、`lock_type`。
    - `lock_type` 至少支持 `global` 与 `per_bus`。
- `parallel_groups`：定义允许并行执行的资源分组。
    - 关键字段：`name`、`resources`、`allow_parallel`。
- 配置要求：同一资源不得同时出现在互斥与并行白名单的冲突路径中。

#### 4.3.3 锁实现方案
**文件锁（跨进程支持）设计要求**：
- 必须支持 `acquire(timeout)` 与 `release()` 两个原子操作。
- 获取失败应返回可识别状态，不应导致调用方阻塞失控。
- 锁文件路径应可配置，并支持按资源维度拆分。
- 释放操作需幂等，防止重复释放导致异常。

#### 4.3.4 死锁预防
- 超时自动释放（默认 30 秒）
- 锁获取顺序：按资源名字典序
- 日志记录：所有锁操作写入 `logs/resource_lock.log`

### 4.4 Case Runner
职责：
1. 执行模块初始化。
2. 逐项调用 Functions。
3. 汇总判定 pass/fail。
4. 输出模块级结果到 `tmp/<module>_result.json`。
5. 写详细日志并可触发报告片段生成。

---

### 4.5 Function Runner
职责：
1. 标准化入参校验。
2. 选择平台适配器执行（Linux/Zephyr）。
3. 解析输出并映射标准状态码。
4. 返回统一结果结构：`code/message/duration/details`。

---

### 4.6 平台适配层

#### 4.6.1 LinuxAdapter
- 命令执行：`execute(command, timeout)`
- 系统日志采集：`collect_syslog()`（dmesg/journalctl）
- 设备检测：`detect_device()`

#### 4.6.2 ZephyrAdapter（简单文本命令协议）

**协议设计**：
- 命令格式：`cmd param1 param2 --flag\n`
- 响应格式：Zephyr shell 标准输出 + 结束标记
- 结束标记：`<<<END>>>` 或超时（默认 5 秒）

**实现要点**：
1. **超时检测**：串口读取超时（默认 5 秒）
2. **行缓冲读取**：逐行读取直到结束标记
3. **响应解析**：提取关键数据，忽略装饰文本
4. **异常处理**：超时/错误响应返回 `DEVICE_ERROR(-102)`

**接口约束**：
- `execute(command, timeout)`：返回标准 `CommandResult`，含成功标志、输出、错误信息与耗时。
- 适配器不得向上层泄露串口实现细节（端口重连、缓存策略等）。
- 协议解析失败时返回统一错误码并附最小可诊断上下文。

#### 4.6.3 适配器统一接口
- `execute(command, timeout)` → `CommandResult`
- `collect_syslog()` → `str`
- `detect_device()` → `Dict`

## 6. 关键流程设计

### 6.1 单次功能测试流程（研发）
1. 用户执行 `run_fixture --name 功能快速验证`。
2. Fixture Runner 加载场景并创建执行上下文。
3. Case Runner 逐个（或并行）执行 Case。
4. Function Runner 执行函数命令并返回标准结果。
5. 写入 tmp、logs；根据配置决定是否写 report。
6. 终端显示汇总结果。

### 6.2 循环测试流程（高低温/EMI/EMS）
1. 根据 `loop_count/loop_interval` 进入循环。
2. 每轮更新循环状态与统计信息。
3. 失败时按 `retry/retry_interval` 执行重试。
4. Dashboard 实时读取 `tmp` 与 `system_monitor.json`。
5. 达到循环次数后输出总体统计与失败清单。

### 6.3 生产测试流程
1. 校验 `sn_required` 与 `--sn`。
2. 顺序执行全功能 Cases（可 `stop_on_failure=true`）。
3. 生成报告并按配置上传 OSS。
4. 输出可追溯信息（SKU、SN、时间戳、结果）。

## 7. 日志、报告与看板设计

### 7.1 四级输出体系
1. 终端：面向 EE/TE，只展示关键状态。
2. `logs/`：面向 SE，保存 debug 细节（1~3级）。
3. `tmp/`：面向 Dashboard，覆盖写最新状态。
4. `reports/`：面向归档与溯源。

### 7.2 Debug 分级策略
- Level 1：平台检测、关键路径。
- Level 2：命令与返回值。
- Level 3：函数调用栈、关键变量、全链路追踪。

### 7.3 报告生成策略
- 生产默认开启，研发可关闭。
- 命名：`{SKU}_{SN|STAGE}_{YYYYMMDD_HHMMSS}_{pass|fail}.report`
- 内容：基础信息、测试项汇总、失败详情、系统信息、统计总结。

### 7.4 OSS 上传接口设计（阿里云）

#### 7.4.1 接口规范
**能力目标**：支持报告归档上传、失败重试、可追溯查询。

#### 7.4.2 配置参数
**字段约束**：
- `endpoint`：OSS Bucket 所在区域 Endpoint
- `bucket`：Bucket 名称
- `auth_type`：鉴权类型（`ak_sk` 或 `sts`）
- `access_key_id/secret`：长期密钥（建议从环境变量读取）
- `upload_timeout`：单次上传超时（秒）
- `retry_count`：失败重试次数（默认 3 次）
- `remote_prefix`：OSS 对象存储路径前缀

#### 7.4.3 上传流程
```
 Fixture 执行完成
    ↓
生成报告文件（reports/ 目录）
    ↓
读取 OSS 配置
    ↓
初始化 OSS Bucket 客户端
    ↓
上传报告（带重试）
    ↓
记录上传结果（成功/失败）
    ↓
输出可追溯 URL
```

#### 7.4.4 可靠性约束
- 上传操作应支持超时控制与指数退避重试。
- 同一报告重复上传应具备幂等处理能力。
- 上传失败不应阻断本地报告落盘与测试结果归档。

#### 7.4.5 安全建议
- **密钥管理**：使用环境变量或密钥管理服务（KMS）
- **权限控制**：Bucket 设置为私有读，授权访问
- **生命周期**：配置 OSS 生命周期规则，自动清理过期报告

### 7.5 Dashboard 设计
#### CLI Dashboard（必须）
- 技术：Python + rich/ncurses。
- 刷新：1~2 秒。
- 快捷键：`Q/R/D/L/S`。
- 自适应：自动识别循环、生产、功能验证模式。

#### Web Dashboard（可选）
- 轻量服务：`--web --port 8080`。
- 数据来源：`tmp/*_result.json` + `system_monitor.json`。
- 刷新：2~5 秒，最多 5 并发客户端。

### 7.6 Schema Discovery
- 启动时扫描 `tmp/*_result.json`。
- 动态解析 JSON 字段并渲染模块卡片。
- 新增模块无需改前端代码即可显示（字段未知时做降级展示）。

### 7.7 日志轮转与清理

#### 7.7.1 轮转策略
**文件大小轮转**：
- 单文件最大：10MB
- 切分命名：`eth_20260222_143022_001.log`、`eth_20260222_143022_002.log`
- 实现：`logging.handlers.RotatingFileHandler`

**时间轮转（可选）**：
- 按天切分：`eth_20260222.log`、`eth_20260223.log`
- 实现：`logging.handlers.TimedRotatingFileHandler`

#### 7.7.2 压缩策略
- 旧日志自动压缩：`.log` → `.log.gz`
- 压缩时机：轮转后立即压缩
- 压缩库：`gzip`

#### 7.7.3 清理策略
| 日志类型               | 保留期限 | 清理方式 |
| ---------------------- | -------- | -------- |
| logs/（研发）          | 7 天     | 定时清理 |
| logs/（生产）          | 30 天    | 定时清理 |
| logs/（失败）          | 永久     | 不清理   |
| tmp/                   | 即时覆盖 | 不清理   |
| reports/               | 30 天    | 定时清理 |
| reports/（OSS 已上传） | 7 天     | 本地删除 |

#### 7.7.4 运维接口约定
- 提供日志清理、报告清理、历史压缩三类运维工具。
- 清理工具应支持按时间窗口执行并输出摘要统计。
- 清理动作应记录到审计日志，便于追溯。

## 9. 部署与环境自更新设计

### 9.1 一键部署（setup_env.sh）
功能：
1. 检测系统类型（Linux/发行版）。
2. 检测 Python/Shell 基础环境。
3. 安装依赖（如 iperf3、串口工具、Python 包）。
4. 初始化目录（logs/tmp/reports）。
5. 输出安装结果摘要。

### 9.2 环境自更新（Manifest 驱动）

#### 9.2.1 依赖清单格式
新增 `functions/requirements_manifest.json` 管理每个函数的依赖，按“函数名 → 依赖项集合”组织。

**字段说明**：
- `system_packages`：需要 apt/yum 安装的系统工具
- `python_packages`：需要 pip 安装的 Python 包（支持版本约束）
- `kernel_modules`：需要加载的内核模块

#### 9.2.2 依赖检查流程
```
Function 执行前
    ↓
加载 requirements_manifest.json
    ↓
检查 system_packages（which 命令）
    ↓
检查 python_packages（import 尝试）
    ↓
检查 kernel_modules（lsmod）
    ↓
发现缺失 → 返回 ENV_MISSING(-2) + 提示修复命令
    ↓
全部通过 → 执行 Function
```

#### 9.2.3 自动修复（可选）
`DependencyChecker` 提供“检测→修复→复检”闭环能力，返回 `(是否全部满足, 缺失列表)`。

**自动修复约束**：
- 必须受全局开关控制（可审计、可回滚）。
- 必须记录修复动作与结果。
- 生产模式默认仅提示，不自动执行高风险变更。

#### 9.2.4 配置位置
- Manifest 文件：`functions/requirements_manifest.json`
- 全局配置：`config/global_config.json` 中 `auto_fix_dependencies` 开关

### 10.4 文档与代码规范
- 配置文件 key 全英文。
- 注释采用 Google 风格，中英双语。
- 提供 Wiki 自动生成脚本（从注释提取）。

## 11. 错误处理与可靠性设计

### 11.1 统一错误处理
- 参数错误：立即失败，返回 `2`。
- 环境缺失：返回 `-2`，触发自更新或提示。
- 设备不可达：返回 `-101/-102` 并附错误详情。
- 执行超时：返回 `1` 并保留超时现场日志。

### 11.2 重试策略
- 支持 Function 级与 Case 级重试（默认 Case 级）。
- 可配置 `retry` 与 `retry_interval`。
- 每次重试写入 `retry_count` 到 tmp/result。

### 11.3 日志保留与轮转
- `logs`：研发 7 天、生产 30 天、失败永久归档。
- 单文件 >10MB 自动切分并压缩 `.gz`。
- `tmp` 覆盖写，不做历史存储。

## 16. MCP 服务层与 AI Agent 集成

### 16.1 MCP 服务层架构

**MCP (Model Context Protocol)** 是一个标准化协议，让 AI 模型能够与外部工具/服务交互。

在本框架中，MCP 服务层作为 AI Agent 与测试函数之间的桥梁：

```
┌─────────────────────────────────────────────────────────┐
│                    AI Agent Layer                       │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │   Skill     │  │ MCP Client   │  │ JSON Generator│  │
│  └─────────────┘  └──────────────┘  └───────────────┘  │
└────────────────────────┬────────────────────────────────┘
                         │ MCP Protocol
                         ▼
┌─────────────────────────────────────────────────────────┐
│                    MCP Service Layer                    │
│  ┌─────────────────────────────────────────────────────┐│
│  │  MCP Server (FastMCP)                               ││
│  │  ├─ Tools (工具)                                     ││
│  │  │   ├─ test_eth()                                 ││
│  │  │   ├─ test_uart()                                ││
│  │  │   └─ test_i2c()                                 ││
│  │  ├─ Resources (资源)                                ││
│  │  │   ├─ 板子信息                                    ││
│  │  │   └─ 测试结果                                    ││
│  │  └─ Prompts (提示词模板)                            ││
│  │      └─ 生成测试配置                                 ││
│  └─────────────────────────────────────────────────────┘│
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│                 现有测试框架层                            │
│  ┌─────────────────┐  ┌───────────────────────────────┐ │
│  │ FunctionRunner  │  │ FixtureRunner / CaseRunner    │ │
│  └─────────────────┘  └───────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 16.2 设计边界与职责

本章节仅定义 MCP 服务层的**架构职责、接口契约、数据约束与安全边界**，不展开具体实现代码与部署命令。

- **MCP Service Layer 职责**：
  1. 将测试函数标准化暴露为可发现的 MCP Tools。
  2. 暴露板卡信息、测试结果等只读 Resources。
  3. 提供配置生成与解释类 Prompts 模板。
  4. 统一返回结构并复用现有状态码。
- **非目标（Out of Scope）**：
  - 大模型策略本身（推理链路、模型选择）。
  - 业务 Web 产品化界面。
  - 驱动级硬件通信细节。

### 16.3 MCP 接口契约（Contract）

#### 16.3.1 Tool 契约（Function → MCP Tool）
| 项 | 约束 |
| --- | --- |
| 命名 | 与 Function 保持语义一致（如 `test_eth`、`test_uart`） |
| 输入 | 显式参数 + 可选默认值；必须参数缺失时返回状态码 `2` |
| 输出 | 统一结构：`code`、`message`、`details` |
| 状态码 | 复用第 5.5 节规范，不新增冲突状态码 |
| 幂等性 | 查询/诊断型工具应尽量幂等，避免副作用 |

#### 16.3.2 Resource 契约（数据暴露）
| 资源类型 | URI 模式 | 返回内容 | 异常约定 |
| --- | --- | --- | --- |
| 板卡信息 | `board://{board_id}/info` | 板卡配置 JSON | 不存在时返回可读错误信息 |
| 测试结果 | `test://{test_id}/result` | `tmp/*_result.json` 对应内容 | 不存在时返回可读错误信息 |

#### 16.3.3 Prompt 契约（模板）
| Prompt | 输入 | 输出 |
| --- | --- | --- |
| 配置生成 | `board_type`、`test_scenario` | 合法 JSON 配置文本 |
| 结果解释（可扩展） | 结果摘要、失败项 | 面向 EE/TE 的解释与建议 |

### 16.4 AI Agent Skill 设计

#### 16.4.1 Skill 职责
Skill 负责“意图到配置”的转换，建议遵循固定四步：

1. **发现能力**：列出可用 Tools/Resources。
2. **选择能力**：根据用户请求匹配最小必要工具集。
3. **生成配置**：产出 Case/Fixture JSON。
4. **验证与纠错**：做 Schema 校验并返回可读错误。

#### 16.4.2 设计要求
- 配置生成失败时，必须返回“字段级错误提示”，而非仅返回失败。
- 未识别场景时，降级到“推荐模板 + 待补参数”模式。
- Skill 不直接绕过调度层执行底层命令，统一经 Runner 链路。

---

### 16.5 JSON Schema 验证策略

- 校验入口：配置生成后、执行前。
- 校验对象：Fixture 与 Case 两类 Schema。
- 校验结果：返回 `(is_valid, error_message)`。
- 错误输出：包含字段路径、期望类型、实际值。
- 版本策略：Schema 与框架版本绑定，避免跨版本歧义。

### 16.7 扩展方向：开放给用户

MCP 用好后，可以考虑开放给用户，用户就能用 AI Agent 去搭他自己的应用了。

#### 16.7.1 用户场景
1. **用户自定义测试**：用户上传板子信息，AI 自动生成测试配置
2. **远程测试服务**：用户通过 Web 界面提交测试请求，后台 AI 执行
3. **测试模板市场**：用户分享测试配置模板

#### 16.7.2 安全考虑
- API 鉴权：JWT Token 或 OAuth2。
- 速率限制：按调用方限流，避免滥用。
- 沙箱环境：用户侧任务隔离执行，限制设备与文件访问范围。
- 审计日志：记录调用方、时间、请求摘要、工具调用与结果。
- 数据分级：板卡敏感信息与生产数据分级脱敏输出。
