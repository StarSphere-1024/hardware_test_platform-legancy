## 1. 设计目标与范围

### 1.1 设计目标
1. 统一 Seeed 主流硬件平台（Nordic、ESP、CM4/CM5、RK、Jetson、TI）的测试调用方式。
2. 提供标准化命令与状态码，降低 EE/TE 使用门槛。
3. 通过 Functions → Cases → Fixtures 三层结构实现高复用、低耦合。
4. 支持研发与生产两类场景：单次测试、循环测试、生产全检、抽检。
5. 提供四级输出体系：终端、详细日志、中间结果、报告，并支持可视化看板。
6. 支持一键部署与环境自更新。

### 1.2 范围（In Scope）
- 抽象层架构与接口规范。
- 配置模型、执行引擎、日志报告、Dashboard 设计。
- SOC（Linux）与 MCU（Zephyr）的统一调用抽象。

### 1.3 非范围（Out of Scope）
- 具体硬件驱动实现细节。
- GUI 复杂交互产品化设计（仅 CLI Dashboard 必须、Web Dashboard 可选）。
- 云端平台完整实现（仅定义上传接口与契约）。

## 2. 总体架构设计

### 2.1 分层架构

```text
┌──────────────────────────────────────────┐
│           Interface Layer                │
│  CLI/API（run_fixture、run_case、函数命令） │
├──────────────────────────────────────────┤
│            Orchestration Layer           │
│  Fixture Runner / Case Runner / Scheduler│
├──────────────────────────────────────────┤
│             Test Logic Layer             │
│  Functions（原子能力）+ Case 编排逻辑     │
├──────────────────────────────────────────┤
│          Platform Adapter Layer          │
│  LinuxAdapter / ZephyrAdapter / CommonIO │
├──────────────────────────────────────────┤
│           Output & Observability         │
│  terminal + logs + tmp + reports + dash  │
└──────────────────────────────────────────┘
```

### 2.2 与需求三层的映射
- Functions：最小可复用原子测试单元。
- Cases：按模块定义完整测试逻辑（初始化/执行/判断/输出）。
- Fixtures：按场景组合多个 Cases，支持串并行、循环、重试、失败策略。

### 2.3 架构关键原则
- 单一职责：Function 不承担编排，Case 不承担跨场景编排。
- 配置驱动：所有行为由 JSON 配置控制。
- 状态驱动：执行状态统一写入 `tmp/*_result.json`。
- 可扩展：新增模块仅需新增 Function + Case，不改核心调度器。

## 4. 核心模块设计

### 4.1 CLI 接口层
- `run_fixture --name <fixture> --config <path> [--sn <SN>]`
- `run_case --config <case_json>`
- `test_xxx`（Function 命令）
- 所有 Function 强制支持：
  - `-h`：帮助
  - `-I`：循环次数
  - `-i`：间隔时间
  - `-r`：开启报告
  - `-w`：超时时间
- 缺少必要参数返回 `2` 并提示错误。

### 4.2 调度引擎（Scheduler）
职责：
1. 加载 Fixture/Case 配置。
2. 根据 `execution` 决定串行/并行。
3. 执行循环、重试、失败处理（`stop_on_failure`）。
4. 维护运行上下文（loop_idx、retry_count、start_time、SN/SKU）。

### 4.3 Case Runner
职责：
1. 执行模块初始化。
2. 逐项调用 Functions。
3. 汇总判定 pass/fail。
4. 输出模块级结果到 `tmp/<module>_result.json`。
5. 写详细日志并可触发报告片段生成。

### 4.4 Function Runner
职责：
1. 标准化入参校验。
2. 选择平台适配器执行（Linux/Zephyr）。
3. 解析输出并映射标准状态码。
4. 返回统一结果结构：`code/message/duration/details`。

### 4.5 平台适配层
- `LinuxAdapter`：命令执行、系统日志采集（dmesg/journalctl）。
- `ZephyrAdapter`：串口命令发送、回包解析、异常栈提取。
- 适配器只暴露统一接口：
  - `execute(command, timeout)`
  - `collect_syslog()`
  - `detect_device()`

### 4.6 输出与可观测层
- 终端输出（Level 0）：简洁状态。
- 日志输出（Level 1~3）：用于调试排障。
- 中间状态：`tmp` 覆盖写。
- 报告输出：`reports/{SKU}_{SN|STAGE}_{ts}_{result}.report`。

## 5. 配置与数据模型设计

### 5.1 全局配置（global_config.json）
```json
{
  "product": {
    "sku": "CM4",
    "stage": "DVT",
    "default_sn_for_test": "DVT",
    "engineer": "张三"
  },
  "runtime": {
    "default_retry": 2,
    "default_retry_interval": 5,
    "default_timeout": 60
  }
}
```

### 5.2 Case 配置模型
关键字段：
- `case_name`, `module`, `functions[]`, `execution`, `timeout`
- 每个 function：`name/params/enabled`

### 5.3 Fixture 配置模型
关键字段：
- `fixture_name`, `cases[]`, `execution`, `stop_on_failure`
- 循环：`loop/loop_count/loop_interval`
- 生产：`sn_required/report_enabled/upload_oss`

### 5.4 中间结果模型（tmp/*_result.json）
```json
{
  "module": "eth",
  "case_name": "生产测试",
  "status": "pass",
  "timestamp": "2026-02-22 14:30:27",
  "duration": 5.2,
  "retry_count": 1,
  "platform": "linux",
  "details": {},
  "error": null
}
```

### 5.5 状态码规范
| 状态码 | 含义                   |
| ------ | ---------------------- |
| 0      | 测试成功               |
| 1      | 超时                   |
| 2      | 缺少参数               |
| -1     | 测试失败（条件不满足） |
| -101   | 找不到设备             |
| -102   | 设备异常               |
| -103   | 未找到指定路径文件     |
| -2     | 缺少环境/依赖包        |

## 6. 关键流程设计

### 6.1 单次功能测试流程（研发）
1. 用户执行 `run_fixture --name 功能快速验证`。
2. Fixture Runner 加载场景并创建执行上下文。
3. Case Runner 逐个（或并行）执行 Case。
4. Function Runner 执行函数命令并返回标准结果。
5. 写入 tmp、logs；根据配置决定是否写 report。
6. 终端显示汇总结果。

### 6.2 循环测试流程（高低温/EMI/EMS）
1. 根据 `loop_count/loop_interval` 进入循环。
2. 每轮更新循环状态与统计信息。
3. 失败时按 `retry/retry_interval` 执行重试。
4. Dashboard 实时读取 `tmp` 与 `system_monitor.json`。
5. 达到循环次数后输出总体统计与失败清单。

### 6.3 生产测试流程
1. 校验 `sn_required` 与 `--sn`。
2. 顺序执行全功能 Cases（可 `stop_on_failure=true`）。
3. 生成报告并按配置上传 OSS。
4. 输出可追溯信息（SKU、SN、时间戳、结果）。

## 7. 日志、报告与看板设计

### 7.1 四级输出体系
1. 终端：面向 EE/TE，只展示关键状态。
2. `logs/`：面向 SE，保存 debug 细节（1~3级）。
3. `tmp/`：面向 Dashboard，覆盖写最新状态。
4. `reports/`：面向归档与溯源。

### 7.2 Debug 分级策略
- Level 1：平台检测、关键路径。
- Level 2：命令与返回值。
- Level 3：函数调用栈、关键变量、全链路追踪。

### 7.3 报告生成策略
- 生产默认开启，研发可关闭。
- 命名：`{SKU}_{SN|STAGE}_{YYYYMMDD_HHMMSS}_{pass|fail}.report`
- 内容：基础信息、测试项汇总、失败详情、系统信息、统计总结。

### 7.4 Dashboard 设计
#### CLI Dashboard（必须）
- 技术：Python + rich/ncurses。
- 刷新：1~2 秒。
- 快捷键：`Q/R/D/L/S`。
- 自适应：自动识别循环、生产、功能验证模式。

#### Web Dashboard（可选）
- 轻量服务：`--web --port 8080`。
- 数据来源：`tmp/*_result.json` + `system_monitor.json`。
- 刷新：2~5 秒，最多 5 并发客户端。

### 7.5 Schema Discovery
- 启动时扫描 `tmp/*_result.json`。
- 动态解析 JSON 字段并渲染模块卡片。
- 新增模块无需改前端代码即可显示（字段未知时做降级展示）。

## 8. 系统监控与性能设计

### 8.1 system_monitor 服务
- 定时采集：CPU、内存、温度、频率、存储。
- 输出：`tmp/system_monitor.json` + `logs/system_monitor_*.log`。
- 场景开关：由 `config.dashboard.system_monitor.enabled` 控制。

### 8.2 性能目标
- CLI Dashboard：CPU < 5%，内存 < 50MB。
- Web Dashboard：页面加载 < 1s，5 并发可用。
- 主测试流程优先级高于监控线程，避免监控影响测试准确性。

## 9. 部署与环境自更新设计

### 9.1 一键部署（setup_env.sh）
功能：
1. 检测系统类型（Linux/发行版）。
2. 检测 Python/Shell 基础环境。
3. 安装依赖（如 iperf3、串口工具、Python 包）。
4. 初始化目录（logs/tmp/reports）。
5. 输出安装结果摘要。

### 9.2 环境自更新
- 启动 Fixture 前执行依赖检查（manifest 驱动）。
- 当新增 Function/Case 声明新依赖时，自动补齐并记录日志。
- 缺依赖时返回 `-2`，并提供自动修复提示。

## 10. 扩展机制与开发规范

### 10.1 新增 Function 标准步骤
1. 在 `functions/<module>/` 实现命令。
2. 完成统一参数与状态码适配。
3. 提供 `README.md` + `README_zh.md`（Google 注释来源）。
4. 在对应 Case 配置中新增 function。

### 10.2 新增 Case 标准步骤
1. 新建 `cases/<module>_case.json`。
2. 定义初始化、执行顺序、超时、判定条件。
3. 本地运行 `run_case` 验证。

### 10.3 新增 Fixture 标准步骤
1. 新建 `fixtures/<场景>.json`。
2. 组合已有 Cases。
3. 设置循环/重试/报告策略。

### 10.4 文档与代码规范
- 配置文件 key 全英文。
- 注释采用 Google 风格，中英双语。
- 提供 Wiki 自动生成脚本（从注释提取）。

## 11. 错误处理与可靠性设计

### 11.1 统一错误处理
- 参数错误：立即失败，返回 `2`。
- 环境缺失：返回 `-2`，触发自更新或提示。
- 设备不可达：返回 `-101/-102` 并附错误详情。
- 执行超时：返回 `1` 并保留超时现场日志。

### 11.2 重试策略
- 支持 Function 级与 Case 级重试（默认 Case 级）。
- 可配置 `retry` 与 `retry_interval`。
- 每次重试写入 `retry_count` 到 tmp/result。

### 11.3 日志保留与轮转
- `logs`：研发 7 天、生产 30 天、失败永久归档。
- 单文件 >10MB 自动切分并压缩 `.gz`。
- `tmp` 覆盖写，不做历史存储。

## 13. 验收标准（与需求对齐）
1. 能通过单命令运行指定 Fixture 并完成多平台测试。
2. Functions/Cases/Fixtures 三层解耦，新增模块无需改核心。
3. 所有 Function 支持 `-h/-I/-i/-r/-w`。
4. 状态码与需求一致，错误可追踪。
5. 循环测试可实时看板展示（CLI 必须）。
6. 生产测试可生成可追溯报告（SKU/SN/时间/结果）。
7. 支持一键部署与新增依赖自动补齐。
